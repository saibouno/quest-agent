\## 概要

この会話では、当初「1年で時価総額1兆円級のものを非エンジニアが作れるか」という話から始まり、  

最終的に \*\*Quest Agent\*\* というプロダクト構想に到達した。



Quest Agent は、単なるToDoアプリではなく、



\*\*大きな目標を、今日進めるクエストに変えて、達成まで並走するエージェント\*\*



として定義される。



内部思想としては \*\*Goal OS\*\* に近いが、外向きの名前は \*\*Quest Agent\*\* とする。

\## ここまでの大きな流れ



\### 1. 非エンジニアでも大きなソフト事業は作れるか

\- 結論として、非エンジニアでも大きな事業の起点は作れる

\- ただし、汎用的な開発基盤そのものを最初から作るのは難易度が高い

\- 非エンジニアの勝ち筋は、コード力そのものではなく

&nbsp; - 現場課題の解像度

&nbsp; - 誰よりも深い痛みの理解

&nbsp; - 速い検証

&nbsp; - 正しい課題設定

\- その観点から、ユーザーにとっては「飲食」「店舗運営」「開業」「実行支援」などの文脈が強みになる



\### 2. 毎日使われるソフトとは何か

毎日使われるソフトは、主に以下のいずれかをやっていると整理した。

\- 今日どう動くか決める

\- 面倒な作業を代わりに進める

\- 状態を見える化する

\- 疲れや迷いを減らす

\- 昨日より前進した実感を出す

という方向性があり、そこから



\- Goal OS

のような構想が生まれた。

とするのがよいという話になった。

\## Quest Agent の定義



Quest Agent は、単なるタスク管理アプリではない。  

以下を含む \*\*目標設定から軌道修正までを支援する並走エージェント\*\* である。



\### Quest Agent が扱うもの

\- 目標設定そのものの壁打ち

\- ゴールの言語化

\- 現在地の整理

\- 制約の整理

\- 期限の整理

\- 勝ち条件（成功条件）の整理

\- マイルストーンへの分解

\- 今日のクエスト提示

\- Blocker（詰まり、不安、未決定事項）の記録

\- Review と reroute（軌道修正）



---



\## Quest Agent の本質

RPGっぽい演出は使ってよいが、本質はゲームではない。  

本質は以下の4つ。



\- 何を目指すか整理できる

\- 今日やることが分かる

\- 詰まりが見える

\- 前に進める



つまり、Quest Agent は



\*\*「現実が前に進む感」を毎日作るためのエージェント\*\*



である。



---



\## Goal Setting / Quest Intake の重要性

この会話の中で大事だと確認されたのは、  

Quest Agent は「今日やることを出す」だけでは不十分であり、  

\*\*最初の入口に Goal Setting / Quest Intake が必要\*\* ということ。



\### Quest Intake でやること

\- 何を実現したいか

\- なぜそれをやるのか

\- いつまでにやるのか

\- どんな制約があるか

\- 何をもって達成とするか

\- いまどこまで進んでいるか

\- 何が不安か

\- 何が未決定か



つまり、Quest Agent は「ゴール設定の壁打ち」から始まる。



---



\## Quest Agent の基本ループ

Quest Agent の運用は、以下のループを想定する。



\### 1. Quest Intake

目標を言語化し、現在地・制約・期限・勝ち条件を整理する



\### 2. Quest Map / World Map

ゴールをマイルストーンに分解し、全体地図を作る



\### 3. Daily Quest

今日やるべきクエストを出す



\### 4. Blocker Management

詰まり、不安、未決定事項、判断保留を記録する



\### 5. Review / Reroute

進捗を振り返り、ルートを修正する

\## Quest Agent の記録思想

\*\*「すべてのステップを記録する必要があるのでは？」\*\*

\- 何を次にやるか

\## 自律エージェント群についての考え方

ユーザーは「時間をあまり割けない場合、先に自律エージェント群を作るのはありか？」と質問した。



結論としては、



\### 基本方針

\*\*最初から自律エージェント群にしない方がよい\*\*



理由:

\- 役割分担設計が重い

\- 状態管理が難しい

\- 受け渡しが複雑

\- 無限ループや暴走を防ぐ必要がある

\- 壊れた箇所の特定が難しい



\### よりよい順番

1\. 人力運用 + AI補助

2\. 単独オーケストレーター

3\. 必要になった役割だけ分離



\### 見せ方と中身

\- 表の顔: 1人のQuest Agent

\- 裏の内部構造: 必要に応じて役割分離



最初からユーザーに複雑なマルチエージェントを見せる必要はない。



---



\## Quest Agent を Quest Agent 自身で作る、という考え方

「Goal OSを作るという目標を、Goal OSを作りながら達成するのはありか？」という問いに対して、

\*\*むしろその進め方が一番強い\*\* という整理になった。



この考え方を Quest Agent に置き換えると、



> \*\*Quest Agent を作るという目標を、Quest Agent の最初の実験台として使う\*\*



という形になる。



\### この進め方が強い理由

\- 未完成の段階から使える

\- 本当に必要な機能だけが残る

\- どこで詰まるかがすぐ見える

\- 最初のデモになる

\- 「自分で使って前進できた」がそのまま価値証明になる

\## 推奨スタック

\### アプリ本体

\- Next.js

\- Vercel

\## Codex / Claude Code に関する整理

会話の中で、Codex と Claude Code をどちらで実装するかが話題になった。



\### 推奨

\- 最初は \*\*Codex 主軸\*\*

\- Claude Code は後から補助で入れる



\### 理由

\- 最初から2系統にすると運用が散る

\- プロンプト資産が割れる

\- どちらに何を任せるかが曖昧になりやすい

\- まずは1系統で前進した方がよい
